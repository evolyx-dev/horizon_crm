# Horizon CRM — Development Instructions

## Project Overview
This is a multi-tenant Travel Agency CRM built on Frappe Framework v15.
App name: `horizon_crm`
Location: repo root (this repo IS the app)

## Architecture
- **Multi-tenancy**: Site-per-tenant architecture (each agency = its own Frappe site)
- **Portal**: Server-rendered Jinja templates in `horizon_crm/www/`
- **Desk**: Standard Frappe Desk for admin/staff operations
- **Docker**: `docker/` for quick-start, `.devcontainer/` for Codespaces

## Key Conventions

### DocType Naming
- All DocTypes prefixed with "Travel" (e.g., Travel Agency, Travel Inquiry)
- Supplier DocTypes: Airline Supplier, Hotel Supplier, Visa Agent, Transport Supplier, Tour Operator, Insurance Provider
- Child DocTypes named descriptively (e.g., Itinerary Day Item, Booking Payment)
- Module name: "Horizon CRM"

### Code Style
- Python: Ruff linter (line-length 110, tabs), pre-commit hooks
- JavaScript: Follow Frappe conventions for client scripts
- Jinja: Extend from standard web.html template
- Always use `frappe.throw()` for user-facing errors
- Use `frappe.whitelist()` for API methods exposed to client

### File Organization
```
horizon_crm/horizon_crm/
├── horizon_crm/doctype/     # All DocTypes
├── api/                      # Whitelisted API methods
├── utils.py                  # Shared utility functions
├── public/                   # Static assets
├── templates/                # Jinja templates
└── www/                      # Portal pages
```

### Testing
- Unit tests in each DocType folder: `test_<doctype>.py`
- E2E tests in `tests/e2e/` using Playwright
- Security tests verify cross-agency isolation
- Run unit tests: `bench --site <site> run-tests --app horizon_crm`

### Docker Development
- All development happens inside Docker containers
- Never install directly on host machine
- Use `docker compose up -d` to start the stack
- Use `docker exec` to run bench commands inside container
